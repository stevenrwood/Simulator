# Tool-table format for the grblHAL simulator

The simulator's 3D view carves the stock as the cutter passes through it. To do this it needs each
tool's **diameter** and **shape** — information that standard g-code and grbl's tool table cannot carry
(grbl stores only a tool offset/radius, never a shape). So the CAM post supplies it as **g-code comment
lines** that the simulator parses and every real controller ignores.

Put these at the **top of the program**, before the first tool change.

## Tool-table comments

```gcode
(TOOL T=1 D=6.35  TYPE=FLAT)
(TOOL T=2 D=3.175 TYPE=BALL)
(TOOL T=3 D=12.7  TYPE=VBIT A=60)
(TOOL T=4 D=1.0   TYPE=VBIT A=30)
(TOOL T=5 D=6.0   TYPE=FLAT)
```

### Grammar

- The comment's **first token must be `TOOL`**, followed by space-separated `KEY=VALUE` tokens.
- Case-insensitive. Unknown keys are ignored. Always write `T=1`, not `T1`.

| Key    | Meaning                              | Notes                                                        |
|--------|--------------------------------------|-------------------------------------------------------------|
| `T`    | Tool number                          | Integer 0–64.                                               |
| `D`    | Cutter **diameter**                  | Millimetres. The carve uses radius = D / 2.                 |
| `TYPE` | Cutter shape                         | `FLAT` (or `ENDMILL`) \| `BALL` \| `VBIT`. Unknown → `FLAT`. |
| `A`    | V-bit **included angle**             | Degrees. Only used when `TYPE=VBIT`.                        |

### Shapes

- `FLAT` / `ENDMILL` — flat-bottom cutter; cells within the radius are cut to the tip Z.
- `BALL` — hemispherical tip; cut depth curves up away from the tool axis.
- `VBIT` — cone; needs `A=<included angle>` (e.g. `A=60` for a 60° V-bit).

## Per-tool-change alternative (end-of-line form)

Instead of a top block, you can tag each tool on its own `M6` line by dropping `T=`. The geometry binds
to the tool being selected:

```gcode
T1 M6 (TOOL D=6.35 TYPE=FLAT)
```

Both forms may be mixed. For a single tool use one or the other; if both set the same tool, the one the
controller sees **last** wins.

## Minimal example header

```gcode
(Generated by <CAM post> - simulator tool table)
(TOOL T=1 D=6.35  TYPE=FLAT)
(TOOL T=2 D=3.175 TYPE=BALL)
(TOOL T=3 D=12.7  TYPE=VBIT A=60)
...
T1 M6
...
```

## Footnote: the native controller tool table is not needed

The build enables grbl's tool table (`N_TOOLS=32`), so `G10 L1 P<n> R<r> Z<z>` works — but it carries no
shape and the simulator's carve never reads it. With an ATC flow that probes each tool on the toolsetter
and applies dynamic TLO (`G43.1`), the native table is unused, so there is normally **no reason to emit
`G10 L1` lines**. Only add them if something on the real machine reads the native table (e.g. `G43 H<n>`
pre-measured offsets or `#5401…`).
